# Remote Cam Preview

## 目标

开发一套可在 iOS 与 Android 之间互联的双端应用：用户使用设备 A 拍照时，设备 B 实时显示 A 的相机取景内容，从而让被拍摄者或协作者能够持续观察构图并即时提出意见，避免反复“拍摄—查看—调整—重拍”。

应用通过 Wi-Fi Aware 建立纯点对点连接，不依赖互联网、云端服务器、用户账号、无线路由器或已加入的 Wi-Fi 网络，也不消耗蜂窝流量。

## 平台与兼容性

- 开发原生 iOS 与 Android 应用，并支持以下三种组合互联：
  - iOS ↔ iOS
  - Android ↔ Android
  - iOS ↔ Android
- iOS 最低版本为 iOS 26，仅支持系统和硬件具备 Wi-Fi Aware 能力的设备。
- Android 仅支持系统和硬件具备 Wi-Fi Aware 能力的设备。
- 首个版本不提供局域网、热点、Wi-Fi Direct、蓝牙或互联网中继等降级连接方案。
- 应用启动时应检测 Wi-Fi Aware 能力与当前可用状态；不支持或当前不可用时，向用户显示清晰原因和必要操作，而不是进入无法完成的配对流程。

## 开发环境

- 项目使用 Nix flake 管理可复现的开发环境，并提交 `flake.nix` 与 `flake.lock`。
- 开发者应可在仓库根目录执行 `nix develop` 进入包含项目所需命令行工具的环境。
- flake 应尽量固定 Android/JVM、协议代码生成、格式化、静态检查、测试及其他跨平台工具的版本，并提供常用的 `checks` 或开发命令。
- Xcode、iOS SDK、签名证书、Provisioning Profile、真机调试支持和其他必须由 Apple/macOS 提供的组件不强行封装进 Nix；flake 应检测或记录这些外部依赖及其最低版本。
- 不在个人机器全局安装项目专用命令行依赖；缺少的通用工具优先通过 flake inputs、`nix develop` 或 `nix run` 提供。

## 设备角色

安装在两台设备上的应用完全相同。在开始会话前，每台设备可选择角色：

- **拍摄端（A）**：打开相机、显示本地取景画面、执行拍照，并向监看端发送实时预览流。
- **监看端（B）**：只显示来自 A 的实时取景内容，不提供遥控快门或相机参数控制。

一次会话必须且只能包含一个拍摄端和一个监看端。选择角色后，本次会话中不动态互换角色；如需互换，结束当前会话并重新选择。

## 配对与连接

- 使用 Wi-Fi Aware 进行设备发现、配对和点对点数据连接。
- iOS 优先采用系统提供的 Wi-Fi Aware 配对界面，避免自行实现设备信任和密钥交换界面。
- Android 端应实现与 iOS Wi-Fi Aware 配对及服务发现流程兼容的交互。
- 首次配对需要用户明确选择并确认另一台设备；系统允许的情况下记住已配对设备，后续可快速重连。
- 连接在 Wi-Fi 层经过认证和加密；应用层协议不得依赖固定 IP 地址、MAC 地址或永久有效的临时 peer handle。
- 会话状态至少包括：未配对、正在查找、正在配对、正在连接、已连接、连接中断和不可用。
- 连接中断时应给出明确状态，并允许用户重试或结束会话。
- 应用不建立账号体系，不接入云端服务，不上传预览、照片、设备信息或使用数据。

## 相机与拍照

- 拍摄必须发生在应用内，不尝试读取或嵌入系统相机 App 的取景画面。
- iOS 使用 AVFoundation 相机管线；Android 优先使用 CameraX，并在设备支持时使用合适的厂商/系统扩展。
- 尽量保留系统及设备厂商提供的自动对焦、自动曝光、自动白平衡、HDR 和静态照片处理能力，使成片效果尽可能接近设备原生相机。
- MVP 只需要默认自动拍照体验，不要求监看端控制快门、镜头、变焦、曝光、对焦、闪光灯或其他参数。
- 拍照由用户在拍摄端 A 上触发。
- 照片使用静态照片捕获管线生成，不从实时 H.265 预览视频中截帧充当成片。
- 成片始终按平台惯例保存到 A 的本地照片库；首次需要时请求照片库写入权限，并妥善处理拒绝授权的情况。
- 本任务不包含录像或音频采集、编码、传输与播放。

## 实时预览

- A 将相机实时取景编码为 H.265/HEVC 视频流并发送给 B。
- 目标帧率为 30 fps。
- 默认目标码率为 10 Mbps。
- 端到端玻璃到玻璃延迟目标低于 200 ms。
- 不以固定的 720p 或 1080p 作为默认传输分辨率。建立会话时，B 应向 A 提供以下显示能力：
  - 屏幕原生像素尺寸；
  - 当前可用于预览的视口尺寸；
  - 屏幕方向；
  - 支持的 HEVC 解码能力与上限。
- A 根据 B 的可用视口、相机传感器/预览宽高比、编码器上限及自身可提供的捕获格式，选择能够最大化利用 B 屏幕的传输分辨率。
- 预览应保持相机画面的正确宽高比，不拉伸变形。默认采用等比居中裁切以最大化铺满 B 的预览区域；如受安全区域或设备形态限制，可保留必要的最小黑边。
- 正确处理设备方向和相机图像旋转，确保 B 看到的画面方向与 A 的取景一致。
- 编解码配置应面向实时传输：避免不必要的重排序与长缓冲，播放器不得为了平滑而累积导致超过延迟目标的大缓冲。
- MVP 首要目标是在良好、稳定的近距离 Wi-Fi Aware 链路上实现原生显示目标分辨率、30 fps、约 10 Mbps 的高清低延迟演示。
- 架构需为后续根据链路状况自动降低码率、帧率和分辨率保留接口，但 MVP 不要求实现复杂的自适应码率算法。允许先实现有限的档位切换或仅收集丢包、吞吐、延迟等指标。

## 成片传输

- B 提供“接收成片”开关，默认开启。
- 开关状态应在会话建立后同步给 A，并在会话期间发生变化时及时更新。
- 开关开启时，A 完成静态照片处理及本地保存后，自动将最终成片文件传给 B；传输的是相机静态照片管线的输出文件，而不是预览流截图。
- B 完整接收并校验文件后，将其保存到 B 的本地照片库。
- 开关关闭时，A 仍正常拍照并在 A 本地保存，但不向 B 发送成片，B 也不创建或保存对应文件。
- 成片传输不应永久阻塞实时预览；应通过独立逻辑通道或优先级控制，使预览低延迟优先于照片文件吞吐。
- 传输过程中应显示简洁状态，并处理取消、连接中断、存储失败和照片库权限被拒绝等情况。
- 不要求 MVP 支持历史照片浏览、批量传输、云同步或跨会话补传。

## 跨平台传输协议

### 总体设计

Wi-Fi Aware 只负责发现、配对和建立安全的 IP 数据路径。应用在该数据路径上使用三条相互独立的逻辑通道：

1. **控制通道：HTTP + WebSocket（TCP）**
2. **实时预览通道：RTP/RTCP over UDP**
3. **成片传输通道：HTTP（TCP）**

不使用 RTMP。RTMP 面向向媒体服务器推流，基于 TCP，丢包时会产生队头阻塞，并通常引入额外播放缓冲；其 HEVC 扩展的跨实现兼容性也不适合作为本项目的核心协议。RTP 则有标准的 HEVC 负载格式，并天然提供序号、时间戳、丢包检测和实时反馈机制。

角色确定后，由拍摄端 A 提供本次会话的服务端点，监看端 B 主动建立控制连接并接收媒体。应用角色与 Wi-Fi Aware 底层的 publish/subscribe 角色不得硬编码为同一概念；任意平台设备都必须能够承担 A 或 B。

所有服务只能绑定到本次 Wi-Fi Aware 数据路径和已配对的对端，不监听普通 Wi-Fi、蜂窝或公网接口。每次会话生成不可预测的临时会话 ID 和访问令牌，断开后立即失效。Wi-Fi Aware 已提供链路层认证和加密；MVP 不自行设计长期密钥体系，但协议结构需为后续增加 TLS/SRTP 保留版本协商能力。

### 控制通道

- B 连接 A 的版本化 HTTP API，并升级到一个长期存在的 WebSocket 双向通道。
- MVP 的控制消息使用带明确 `type`、`requestId`、`protocolVersion` 和负载字段的 UTF-8 JSON，优先保证跨平台调试便利；未来可在版本协商后增加 CBOR 等二进制编码。
- HTTP API 路径以 `/v1/` 开头；WebSocket 可使用 `/v1/events`。
- 控制通道至少承载：
  - 协议版本、角色和设备能力协商；
  - B 的原生屏幕像素、可用预览视口、方向及 HEVC 解码上限；
  - 协商后的编码分辨率、宽高比、帧率、码率、HEVC profile/level；
  - RTP/RTCP 地址、端口、payload type 和随机 SSRC；
  - 开始、停止和重新配置预览；
  - B 的“接收成片”开关；
  - 拍照完成、成片可用、传输完成及失败事件；
  - 心跳、错误、断连和会话结束；
  - B 请求关键帧，以及未来的手动或自动降档指令。
- 消息必须有大小上限、超时、重复请求处理和明确错误码。未知的可选字段应被忽略；不兼容的主协议版本应明确拒绝。
- WebSocket 断开即视为会话控制失效；预览与文件传输应尽快停止，避免产生无人管理的后台流量。

### 实时预览通道

- 使用 RTP over UDP 承载 H.265/HEVC 码流，并按照 RFC 7798 对 HEVC NAL unit 进行单包、聚合或分片封装。
- RTP 时钟频率使用视频标准的 90 kHz；序列号、时间戳和 marker bit 必须按帧正确生成，使 B 可检测丢包、重排包并识别帧边界。
- UDP 包大小应根据实际数据路径 MTU 控制，避免 IP 分片；过大的 NAL unit 使用 RFC 7798 Fragmentation Unit 拆分。
- 编码器采用面向实时的配置：不使用 B 帧，限制参考帧和 GOP 长度，避免编码重排序，并周期性产生 IDR 帧。
- 会话开始、编码参数改变及 IDR 前应发送解码所需的 VPS/SPS/PPS。分辨率改变必须通过控制通道通知 B，并以新的参数集和 IDR 作为切换边界。
- B 使用有严格上限的小型重排/抖动缓冲；迟到且已无法按时显示的帧直接丢弃，不通过持续增大缓冲追求无卡顿。
- 使用 RTCP 从 B 向 A 报告接收统计、丢包、抖动和往返时间，并支持 PLI/FIR 类关键帧请求。MVP 可只记录这些指标和触发关键帧，不要求完成连续自适应算法。
- 视频丢包不得阻塞后续帧；必要时丢弃受损帧并请求新关键帧，以延迟优先于逐帧完整性。

### 成片传输通道

- A 完成拍照处理后生成一个仅在当前会话有效的 `photoId`，并通过 WebSocket 发送成片元数据，包括文件名、MIME 类型、字节数、像素尺寸和 SHA-256 摘要。
- 只有 B 的“接收成片”开关处于开启状态时，A 才发布该资源并发送 `photo.available` 事件。
- B 收到事件后通过 `GET /v1/photos/{photoId}` 从 A 拉取成片。由 B 主动拉取可避免要求两端同时运行 HTTP 服务，也能自然遵循 B 的接收开关。
- B 必须先写入临时文件，校验长度与 SHA-256 后再保存到照片库，并通过 WebSocket 返回成功或失败结果。
- 资源在成功确认、会话结束或超时后删除；MVP 不要求断点续传。如文件传输失败，可以在资源有效期内进行有限次数的整文件重试。
- HTTP 文件响应采用流式读写，禁止将整张照片一次性复制到内存。
- 控制、预览与成片传输使用独立连接/通道；调度上始终优先保证控制与实时预览，照片传输不得挤占到使预览延迟明显超过目标。

### 协议实现要求

- 协议定义应单独存放并版本化，避免 Swift 与 Kotlin 实现各自演化出不兼容行为。
- 对 JSON schema、RTP HEVC packetizer/depacketizer、状态机和文件校验逻辑建立共享测试向量；两端使用同一组输入验证编码与解析结果。
- 任何网络输入都必须经过长度、范围、状态和资源限制检查。畸形控制消息、RTP 包或照片元数据不得导致崩溃、无限分配或无限等待。
- 先以独立测试数据验证 iOS/Android 的 TCP、WebSocket、UDP/RTP 双向通信，再接入真实相机编码器和照片库。

## MVP 范围

MVP 必须实现：

1. 两台受支持设备通过系统引导完成 Wi-Fi Aware 配对。
2. 两端选择互补角色并建立纯 P2P 会话。
3. A 在应用内打开相机并本地拍照。
4. B 以保持宽高比且尽量铺满屏幕的方式显示 A 的 HEVC 实时预览。
5. 良好链路下达到 30 fps、默认约 10 Mbps，并以低于 200 ms 延迟为优化目标。
6. A 拍照后始终本地保存；B 开启接收时自动接收并本地保存最终成片，关闭时不传输、不保存。
7. 正确呈现权限、能力不足、连接过程、断连和传输失败状态。
8. iOS 与 Android 交叉组合至少完成一次真机端到端验证。

MVP 不包含：

- 互联网、服务器、账号或云同步；
- 不支持 Wi-Fi Aware 设备的连接降级方案；
- B 端遥控拍摄或调整相机参数；
- 音频、录像或视频文件保存；
- 多拍摄端、多监看端或群组会话；
- 复杂的连续自适应码率算法；
- 历史相册、编辑、批量传输或断点续传。

## 验收标准

- 在无互联网、无路由器、未加入 Wi-Fi 网络的环境中，仅开启两台设备的 Wi-Fi 无线功能即可完成配对、连接和预览。
- iOS 与 Android 均可承担 A 或 B 角色，并且选择角色前两端功能对等。
- 至少验证 iOS → Android 与 Android → iOS 两个方向；条件允许时同时验证同平台组合。
- 良好近距离链路下，B 稳定显示 30 fps 的 HEVC 预览，默认目标码率约 10 Mbps，分辨率根据 B 的原生显示视口和相机宽高比选择，而非固定为 720p。
- 使用高速运动或同时拍摄计时器画面的方式测量端到端延迟，并以低于 200 ms 为通过目标；测试方法与结果需记录。
- 旋转设备后画面方向正确、比例正确且无拉伸，预览尽量铺满 B 的可用显示区域。
- A 按下快门时，生成的照片来自静态照片处理管线并成功保存到 A。
- B 的接收开关开启时，最终成片自动传输且校验一致，并成功保存到 B；关闭时 B 不收到也不保存照片。
- 传输成片期间实时预览保持可用，不因文件传输产生明显的长时间冻结或不可接受的延迟增长。
- 对不支持 Wi-Fi Aware、Wi-Fi 未开启、权限被拒绝、配对失败、断连及保存失败均有明确且可恢复的用户提示。

## 开发与测试注意事项

- Wi-Fi Aware、相机编解码及照片库能力必须使用真机测试；模拟器仅用于不依赖硬件的界面和协议单元测试。
- 在开始完整产品开发前，优先制作一个最小跨平台技术验证：完成 iOS/Android Wi-Fi Aware 互相发现、配对、建立数据通道，并传输可验证的测试数据。跨平台链路验证通过后再接入相机和 HEVC。
- 记录各测试设备型号、系统版本、Wi-Fi Aware 能力、支持的相机输出格式、HEVC 编解码上限以及实际性能数据。
- 协议解析、状态机和文件完整性校验应具备自动化测试；媒体延迟、画质、方向切换和跨平台互联应具备可重复的真机测试步骤。
